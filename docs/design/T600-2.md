# T600-2: Rerun comparison model for before-after repair evidence

## 目的
PLAN.v2.md §25 / Phase 4 のうち、temporary patch 適用後に baseline run と rerun の結果を比較し、修復判断に使う証跡を保存・取得できる backend model/API を追加する。

## スコープ
- 対象: repair rerun 起動 API、before/after comparison schema、comparison 生成ロジック、baseline run 配下への comparison JSON 永続化、取得 API、単体/API テスト。
- 非対象: diff review UI、approve/reject 状態管理、patch の最終適用/破棄 UI。T600-3 で扱う。

## アプローチ
既存 RunManager の run lifecycle と artifact 生成を再利用し、baseline run の `requested` をそのまま使って rerun を起動する。rerun の成果物は既存 top-level run として保存し、baseline run 配下の `reruns/<rerunId>/comparison.json` に比較証跡を保存する。comparison は test summary と failedTests をキー化して resolved / remaining / new failure を算出し、`fixed` / `improved` / `unchanged` / `regressed` / `inconclusive` の verdict を返す。

## 検討した代替案
- RunManager に nested runDir を導入: PLAN の `reruns/<rerunId>/...` に近いが、既存 artifact lifecycle への影響が大きいため不採用。今回は comparison artifact を baseline 配下に保存し、実行 artifact は既存 run model を参照する。
- rerun 完了まで HTTP request を保持: 長時間実行でタイムアウトしやすいため不採用。既存 run API と同じ 202 + polling 方式にする。

## 影響範囲
- 変更ファイル: `packages/shared/src/index.ts`, `apps/agent/src/repair/repairComparison.ts`, `apps/agent/src/routes/runs.ts`, 関連テスト。
- 既存機能への影響: 既存 `/runs` start/list/detail の behavior は変更しない。repair rerun endpoint から既存 RunManager を呼び出すだけに留める。
- マイグレーション要否: なし。comparison JSON は新規 artifact として追加される。

## テスト方針
- 追加するテスト: comparison verdict / resolved / new failures の算出、comparison path の runId validation、repair rerun API の 202 応答、comparison 未生成時 409、保存済み comparison の取得。

## リスクと緩和策
- 意図しない別 project rerun: `loadRun` と current projectStore を経由し、baseline run の project と current project を一致させる。
- path traversal: rerunId path segment は既存 runId 形式に限定してから comparison path を作る。
- 長時間 rerun: HTTP は 202 で即時返却し、比較保存は `handle.finished` の background continuation に限定する。

## 想定外の判断ポイント
PLAN.v2 は `runs/<runId>/reruns/<rerunId>/...` を示すが、既存 RunManager は runDir を固定生成している。T600-2 では nested artifact lifecycle へ踏み込まず、baseline 配下には comparison JSON のみを置く。将来 nested run storage が必要なら RunManager の run path injection を別タスクで扱う。

## セルフレビュー結果
- `/pr-review-toolkit:review-pr` は本環境で利用不可のためフォールバックを採用。
- ユーザー視点: rerun は 202 + polling の既存 run 体験に合わせ、comparison 未生成時は 409 で明示する。
- 運用視点: rerun artifact は既存 run storage に任せ、baseline 配下には軽量な comparison JSON を保存して証跡リンクを残す。
- 攻撃者視点: path に入る rerunId は `run-...-xxxxxxxx` 形式に限定し、任意 path segment を comparison artifact lookup に使わない。
