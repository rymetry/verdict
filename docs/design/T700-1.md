# T700-1: QA understanding inventory metadata model

## 目的
PLAN.v2 Phase 5 の「テスト意味、操作、期待結果、locator/assertion、Allure metadataをQA向けに表示する」の土台として、Test Inventory API に QA 向け metadata の安定したモデルを追加する。

## スコープ
- 対象: shared schema の `TestCase` 拡張、Playwright list JSON inventory からの metadata 生成、API/client 互換テスト。
- 非対象: QA View 表示変更、ts-morph による `test.step` / `expect` / locator 抽出、Allure metadata 抽出。

## アプローチ
既存の `playwright test --list --reporter=json` を引き続き権威ソースにする。T700-1 では list JSON から確実に取得できる describe path + title を `fullTitle` と QA 目的文に正規化し、手順・期待結果は空配列として表現する。抽出元と信頼度を metadata に持たせ、T700-2 は UI 表示、T700-3 は AST/Allure 補強として同じモデルへ追記できる形にする。既存 endpoint の URL や実行方法は変えない。

## 検討した代替案
- T700-1 で ts-morph を導入する: T700-3 の spike 範囲を先取りし、変更規模と設計判断が増えるため不採用。
- QA 表示まで同時に実装する: T700-2 と関心が分かれており、PR 粒度を超えるため不採用。

## 影響範囲
- 変更ファイル: `packages/shared/src/index.ts`, `apps/agent/src/project/inventory.ts`, inventory 関連テスト
- 既存機能への影響: 既存 inventory の file/title/line/tags は維持し、追加 field のみ増やす。
- マイグレーション要否: なし。永続保存済み inventory はなく、API response schema の追加のみ。

## テスト方針
- `parsePlaywrightListJson` が nested describe から `fullTitle` と QA metadata purpose/source/confidence を生成することを確認する。
- shared schema が QA metadata の必須形を検証することを確認する。
- Web client 側の inventory fixture を更新し、既存一覧表示が壊れないことを確認する。

## リスクと緩和策
- UI が未抽出の手順/期待結果を誤解する: `source` と `confidence` を持たせ、T700-2 で表示時に区別できるようにする。
- 既存 fixture の型崩れ: 必要なテスト fixture を更新し、schema parse と typecheck で検出する。
- 過剰抽出: T700-1 では list JSON で確実な情報だけに限定する。

## 想定外の判断ポイント
PLAN.v2 は T700-1 の具体 field 名を指定していない。既存 `TestStepSchema` がすでにあるため、手順と期待結果は同じ構造を再利用し、後続の AST 抽出で line を付与できるようにする。

## セルフレビュー結果
- 要件: Test Inventory に QA 向けの `fullTitle` と `qaMetadata` が追加され、目的・手順・期待結果の器が API schema で検証される。
- スコープ: 表示変更と AST/Allure 抽出は T700-2/T700-3 に残し、T700-1 はモデルと list JSON 由来の初期値に限定した。
- セキュリティ: 新しい入力境界はなく、既存 Playwright list JSON の parse 結果を schema 化して返すだけに限定している。
- 運用: `source` と `confidence` を持たせ、未抽出の steps/expectations を後続 UI が区別できるようにした。
- 一時ファイル: 追加なし。
