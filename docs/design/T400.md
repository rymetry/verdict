# T400: Bun feasibility spike report

## 目的
PLAN.v2 §30 / §31 / §32 の Phase 1.5 を完了し、Bun をユーザー側 package manager として標準対応に進めるべきかを判断する。

## スコープ
- 対象: Bun lockfile / `bunx --no-install --bun` / Playwright CLI / Allure CLI / stdout/stderr streaming / artifact 生成に関する実現性判断
- 非対象: BunCommandRunner の production 実装、CI matrix 追加、Bun fixture の依存 install

## アプローチ
公式 Bun docs とローカル実行結果を突き合わせる。既存の `PackageManagerDetector` は Bun を検出しても `experimental-bun` として実行ブロックしているため、その判断が妥当かを先に検証する。bare `bunx` は npm auto-install と global cache write を伴うため、今回の検証では実行せず、`--no-install` 経路だけを実測する。

## 検討した代替案
- BunCommandRunner を即実装する: `bunx --no-install` が pnpm workspace fixture の local binary を解決できず、artifact pipeline の受け入れ条件を満たせないため不採用。
- bare `bunx playwright` を実行して比較する: 公式 docs 上 auto-install fallback があり、ネットワークと global cache への副作用があるため不採用。

## 影響範囲
- 変更ファイル: `docs/design/T400.md`, `docs/operations/bun-feasibility-report.md`, `IMPLEMENTATION_REPORT.md`
- 既存機能への影響: なし
- マイグレーション要否: なし

## テスト方針
- `bun --version`
- `bunx --no-install --bun playwright --version`
- `bunx --no-install playwright --version`
- `bunx --no-install --bun allure --version`
- docs-only 変更後に `pnpm typecheck` / `pnpm test` で既存機能の回帰がないことを確認する。

## リスクと緩和策
- Bun の仕様変更: report に確認日と公式 docs 参照を残し、将来再評価できるようにする。
- pnpm fixture での失敗を Bun 全体の不可と誤解する: 「Bun-installed project での専用 fixture 検証が次条件」と明記する。
- auto-install による依存汚染: bare `bunx` は実行しない。

## 想定外の判断ポイント
既存 fixture は pnpm workspace で install されているため、Bun の local binary resolution を代表しない。今回の結論は「Bun 標準対応はまだしない」であり、「Bun が不可能」という判断ではない。
