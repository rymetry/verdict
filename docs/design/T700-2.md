# T700-2: QA View purpose and expectation presentation

## 目的
PLAN.v2 Phase 5 / Frontend Design の QA View 要件のうち、Test Inventory にあるテスト目的・操作ステップ・期待結果を QA が読める形で表示する。

## スコープ
- 対象: `TestInventoryPanel` の test row 表示、QA metadata 表示テスト。
- 非対象: metadata 抽出精度の向上、locator/assertion/Allure metadata 抽出、Developer View 表示、Run 実行導線。

## アプローチ
T700-1 で追加済みの `qaMetadata` を source of truth とし、既存 inventory API contract は変更しない。各 test row は title/line/tags の下に purpose を表示し、steps/expectations がある場合だけ短いリストで表示する。low confidence かつ title と重複する purpose は表示せず、metadata source を小さく表示することで、未抽出であることを UI が過度に断定しないようにする。
既存の Card/Badge/Tailwind token を使い、QA View の 3-column density を崩さない。

## 検討した代替案
- 新しい QA detail panel を追加する: T700-2 の要求は inventory 表示で足り、画面構成変更が大きいため採用しない。
- API に表示用 field を追加する: 既存 `qaMetadata` で表現できるため採用しない。

## 影響範囲
- 変更ファイル: `apps/web/src/features/test-inventory/TestInventoryPanel.tsx`, `apps/web/test/features/test-inventory-panel.test.tsx`
- 既存機能への影響: inventory 取得・run 実行・failure review は変更しない。
- マイグレーション要否: なし。

## テスト方針
- purpose、steps、expectations、metadata confidence/source が表示されること。
- steps/expectations が空の既存 low-confidence metadata でも表示が破綻せず、title と重複する purpose が二重表示されないこと。
- 既存の blocked/loading/error/empty/list 表示を維持すること。

## リスクと緩和策
- 表示密度増加: `details` ではなく常時表示する項目を purpose 中心に絞り、steps/expectations は存在時のみ短い grid にする。
- 誤解を招く断定表示: source/confidence badge を添えて heuristic / low-confidence であることを示す。
- UI regression: 既存テストに QA metadata の期待を追加し、routing/layout には触れない。

## 想定外の判断ポイント
T700-3 の抽出強化前は steps/expectations が空のテストが多いため、T700-2 では空 state を個別 test に大量表示しない。source/confidence の表示だけで「抽出済みの範囲」を示す。
