# T1000-2: CI artifact import for Playwright and Allure outputs

## 目的
Phase 8 のレビュー導線で、GitHub Actions などの CI artifact metadata から Playwright / Allure / QMO 関連 artifact link を構造化して取り込めるようにする。

## スコープ
- 対象: CI artifact import 用 shared schema、artifact name classification、run-scoped import API、web API client。
- 非対象: GitHub API からの artifact 取得、artifact ZIP の download/extract、UI 表示、PR comment 送信。これらは認証・外部通信・UI の責務を含むため後続に分離する。

## アプローチ
T1000-1 の `CiArtifactLink` を拡張し、CI から渡された artifact metadata を Workbench が扱う既知 kind に正規化する。名前から Playwright report/results、Allure report/results、Quality Gate、QMO summary、log を分類し、未対応 artifact は skipped として返す。URL は既存の HTTP(S) 境界を流用し、外部 artifact への接続やローカルファイル書き込みは行わない。

## 検討した代替案
- GitHub Actions API を直接呼ぶ: 認証情報、rate limit、artifact download の扱いが必要になり T1000-2 の範囲を超えるため不採用。
- artifact ZIP を展開して run directory に保存する: zip bomb / path traversal / retention の設計が必要で、現タスクでは link import に限定する。

## 影響範囲
- 変更ファイル: packages/shared/src/index.ts, apps/agent/src/reporting/ciArtifactImport.ts, apps/agent/src/routes/runs.ts, apps/web/src/api/client.ts, 関連テスト
- 既存機能への影響: 既存 run/QMO/Release Review draft の挙動は変更しない。
- マイグレーション要否: なし。

## テスト方針
- shared schema が import request/response を検証することを確認する。
- import model が Playwright/Allure/QMO artifact を分類し、未対応 artifact を skipped に分けることを確認する。
- API が run に紐付けた import response を返し、不正入力を 400 にすることを確認する。
- web client が import endpoint を呼び、response schema を検証することを確認する。

## リスクと緩和策
- artifact 名の表記揺れ: lowercase 正規化と複数の既知 token で分類する。未知 artifact は `other` に落とさず skipped として明示する。
- URL/外部通信リスク: URL は HTTP(S) validation のみ。download/extract は行わない。
- 過剰実装: persistence と UI は追加せず、後続タスクが呼べる API と model に限定する。

## 想定外の判断ポイント
PLAN.v2 は「CI artifact import」と記載するが、認証済み GitHub artifact download は破壊的ではないものの外部要因が大きい。T1000-2 では metadata import に限定し、実体取得は明示タスクが必要と判断した。

## セルフレビュー記録
