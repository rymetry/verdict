# T1200: ReportPortal re-evaluation decision record and provider extension plan

## 目的
PLAN.v2 Phase 10 の「Allure継続、ReportPortal追加、並行運用の判断材料が揃う」を満たすため、ReportPortal を初期採用しない理由、再評価条件、将来の `ReportProvider` 拡張手順を運用判断として残す。

## スコープ
- 対象: ReportPortal 再評価 decision record、将来の `ReportPortalProvider` / remote reporting 拡張計画、既存 `ReportProvider` 境界との整合確認。
- 非対象: ReportPortal 連携コード、credential 設定、remote API 呼び出し、ReportPortal agent 導入、Allure からの移行。

## アプローチ
PLAN.v2 §11/§16/Phase 10 と既存 `ReportProvider` 実装を読み、現在の file-first / Allure-first 方針を維持する判断を文書化する。ReportPortal は中央 triage、権限、Issue Tracker 連携、横断 dashboard が必要になった時に追加する。将来実装では Allure を置き換えず、remote 送信と remote summary 読み取りを provider 境界で段階導入する。

## 検討した代替案
- すぐに ReportPortal adapter を実装する: 現時点の完了条件は判断材料の整備であり、remote credential と運用基盤を増やすのは YAGNI のため採用しない。
- Allure から完全移行する: PLAN.v2 が「完全移行しない」と明記しており、既存の静的アーカイブ・history JSONL・QMO summary を捨てる必要がないため採用しない。

## 影響範囲
- 変更ファイル: `docs/operations/reportportal-re-evaluation.md`, `docs/design/T1200.md`
- 既存機能への影響: なし。文書のみ。
- マイグレーション要否: なし。

## テスト方針
- Markdown のリンク・見出し・スコープが PLAN.v2 と矛盾しないことをレビューする。
- `rg` で ReportPortal 関連記述の整合を確認する。

## リスクと緩和策
- 将来実装時に provider と publisher の責務が混ざる: decision record で read path と write path を分けて記録する。
- 機密情報扱いの見落とし: token / endpoint / project name を設定境界で扱い、ログ・PR コメントに露出しない方針を明記する。
- Allure と ReportPortal history の二重管理: 完全統合しない前提と照合 key 方針を明記する。

## 想定外の判断ポイント
ReportPortal は remote API / agent 前提であり、現行 `ReportProviderInput` は local run artifact 読み取りに寄っている。将来の送信は `ReportProvider.readSummary` に詰め込まず、run 後 publish 経路を別責務として追加する方が既存の read model を壊さない。
