# T1000-1: GitHub PR Issue and CI artifact draft model

## 目的
PLAN.v2 Phase 8 の「PR/Issue/CI artifact/Allure report を紐付けたレビュー導線」の土台として、Release Readiness の PR comment draft に使う入力・出力 schema と draft 生成 API を追加する。

## スコープ
- 対象: shared schema、agent の draft 生成ロジック、run-scoped draft API、web API client。
- 非対象: GitHub への実送信、CI artifact の自動 import、UI 表示。これらは T1000-2/T1000-3 の責務。

## アプローチ
既存の QMO summary を source of truth とし、呼び出し側が渡す PR/Issue/CI artifact link を添えて Markdown draft を生成する。外部サービスへは接続せず、HTTP(S) URL と GitHub metadata を schema validation するだけに留める。QMO summary が未生成の場合は既存 endpoint と同じく 409 とし、draft の根拠が欠けた状態を許可しない。

## 検討した代替案
- GitHub API 連携まで実装する: 認証・権限・送信操作が絡み、T1000-1 の「draft model」を超えるため不採用。
- qmo-summary.md をそのまま返す: PR/Issue/CI artifact link を structured に扱えず、後続 UI/import で再利用しにくいため不採用。

## 影響範囲
- 変更ファイル: packages/shared/src/index.ts, apps/agent/src/reporting/releaseReviewDraft.ts, apps/agent/src/routes/runs.ts, apps/web/src/api/client.ts, 関連テスト
- 既存機能への影響: 既存 run/QMO/repair API の挙動は変更しない。
- マイグレーション要否: なし。

## テスト方針
- shared schema が HTTP(S) URL のみ受け付け、非 HTTP(S) URL を拒否することを確認する。
- draft builder が QMO summary と PR/Issue/CI artifact link から安定した Markdown を生成することを確認する。
- API が QMO summary 欠落時に 409、存在時に draft を返すことを確認する。
- web client が draft endpoint を呼び、409 を pending として扱うことを確認する。

## リスクと緩和策
- URL/Markdown 注入: URL は HTTP(S) に限定し、表示 text は改行を潰して Markdown の構造を壊さない。
- 早すぎる GitHub 依存: 送信・認証・import は行わず、draft 生成だけに閉じる。

## 想定外の判断ポイント
PLAN.v2 は Phase 8 を大枠で記述しており、T1000-1 の詳細は PROGRESS.md の分割名から推定した。CI artifact import は T1000-2、PR comment UI は T1000-3 に残す。
