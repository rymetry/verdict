# T500-3: AI Analysis View for validated analysis results

## 目的
PLAN.v2.md Phase 3 / §26 の validated AI analysis result を QA View で確認できるようにし、原因分類・根拠・リスク・patch 有無を人間がレビューできる入口を作る。

## スコープ
- 対象: Web API client、AI Analysis UI component、QA route への配置、UI テスト。
- 非対象: patch 適用 UI、rerun comparison、AI 結果の履歴一覧、Codex/Gemini adapter 切替。後続タスクで扱う。

## アプローチ
既存 QA View の右列に `FailureReview` と並ぶ `AiAnalysisPanel` を追加する。AI CLI は外部コスト・認証・時間が絡むため自動実行せず、active run があるときだけ明示ボタンで起動する。成功時は `AiAnalysisResponseSchema` で validated された結果だけを表示し、patch は直接適用せず「提案あり」としてコードブロックに留める。

## 検討した代替案
- FailureReview 内に統合: 責務が肥大化し、AI 実行の mutation 状態と failure review query が混ざるため不採用。
- route を新設: T500-3 の目的は validated result の表示であり、既存 QA context から離す必要がないため不採用。

## 影響範囲
- 変更ファイル: `apps/web/src/api/client.ts`, `apps/web/src/features/ai-analysis/AiAnalysisPanel.tsx`, `apps/web/src/routes/qa.tsx`, 関連テスト。
- 既存機能への影響: QA View の右列にパネルが 1 つ増える。Run 実行・Failure Review の query contract は変更しない。
- マイグレーション要否: なし。

## テスト方針
- 追加するテスト: runId なしの empty state、実行中 state、成功時の classification/evidence/risk/files/patch 表示、エラー表示、API client の schema parse。

## リスクと緩和策
- 意図しない AI 実行: 自動 fetch ではなく button click の mutation に限定する。
- AI 出力の過信: confidence と requiresHumanDecision を常時表示し、patch は直接適用しない。
- UI の過密化: 既存カード/Badge/Button primitives を使い、右列内で縦積みにする。

## 想定外の判断ポイント
T500-2 は POST API のみを提供しており、persisted JSON の GET endpoint はまだない。T500-3 はまず「実行して返ってきた validated result を表示する」範囲に限定し、履歴取得は後続に回す。
