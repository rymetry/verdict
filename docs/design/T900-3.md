# T900-3: StorageState and auth setup risk surfacing

## 目的
PLAN.v2 Phase 7 のうち、storageState と auth setup の保守・機密情報リスクを read-only に可視化できる API contract を追加する。

## スコープ
- 対象: config summary の auth risk schema、Agent の heuristic 抽出、Web API client validation、関連テスト。
- 非対象: storageState JSON / cookie / localStorage の本文読み取り、auth setup の実行、config / fixture / POM の編集 UI、POM/locator explorer UI。

## アプローチ
既存 `GET /projects/:projectId/config-summary` に `authRisks` を追加する。Playwright config は引き続き import/evaluate せず、コメント除去済みテキストから `storageState` string/object、`globalSetup` string を検出する。source scan は T900-1 と同じ directory/file caps と skip directory 方針を使い、auth setup らしい file 名や `storageState` を含む setup-like source を signal として返す。
storageState file の中身は読まず、absolute/traversal path は path 自体を返さない高リスク signal として扱う。

## 検討した代替案
- storageState JSON を parse して cookie/token の有無を判定する: PLAN.v2 の「渡さないもの」に反し、機密情報を扱うため採用しない。
- Playwright config を実行して正確な resolved config を得る: ユーザーコード実行と副作用のリスクがあるため採用しない。

## 影響範囲
- 変更ファイル: `packages/shared/src/index.ts`, `apps/agent/src/project/configSummary.ts`, `apps/agent/test/configSummary.test.ts`, `apps/agent/test/sharedSchema.test.ts`, `apps/web/test/api/client.test.ts`
- 既存機能への影響: 既存 config summary response に backward-compatible field を追加する。既存 run や artifact pipeline は変更しない。
- マイグレーション要否: なし。

## テスト方針
- config の `storageState`, inline object, `globalSetup` から risk signal が抽出されること。
- auth setup-like source file が project-relative path だけで返ること。
- unsafe absolute/traversal storageState path は path を返さず high severity になること。
- shared schema と Web API client が `authRisks` を validate すること。

## リスクと緩和策
- 機密情報露出: storageState file 本文は読まない。unsafe absolute path は path 値を response に含めない。
- false positive/negative: source は heuristic として `source: "heuristic"` を明示し、UI が断定表示しない contract にする。
- 大規模 repo の走査負荷: T900-1 と同じ directory/file/size caps と skip directories を使う。

## 想定外の判断ポイント
T900-2 の UI が未完了のため、T900-3 はまず config summary API に risk contract を追加し、UI 表示は T900-2 の explorer 側で同じ field を利用できる状態にする。
