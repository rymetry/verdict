# T500-1: AI analysis shared schema and redacted context builder

## 目的
PLAN.v2 §26 / §31 / §32 の Phase 3 前半として、AI CLI に渡す前の失敗分析 context を型で固定し、機密情報と絶対 path を抑制した形で生成できるようにする。

## スコープ
- 対象: shared zod schema、agent 側 AI analysis context builder、単体テスト
- 非対象: AI CLI 実行、AI 分析 API、GUI 表示、patch 適用、source snippet 読み取り

## アプローチ
既存の `RunMetadata` と T300 の `FailureReviewResponse` を source とし、stdout/stderr は末尾だけを cap して取り込む。すべての自由記述文字列は既存 `redact()` を通し、project root / home directory は placeholder に置換する。file path は project-relative にできるものだけを context に載せ、絶対 path は drop する。AI 出力 schema も shared に追加し、次タスク T500-2 の adapter validation で再利用する。

## 影響範囲
- 変更ファイル: `packages/shared/src/index.ts`, `apps/agent/src/ai/analysisContext.ts`, `apps/agent/test/aiAnalysisContext.test.ts`, `apps/agent/test/sharedSchema.test.ts`
- 既存機能への影響: なし。新しい builder はまだ route に接続しない。マイグレーション不要。

## テスト方針
- schema parse、failure enrichment、path/token redaction、log tail cap を単体テストで固定する。

## リスクと緩和策
- AI context への secret 混入: 既存 redaction と path placeholder 化、bounded log tail、artifact 本文非読込で緩和。
- 失敗原因分析に必要な情報不足: T500-1 は安全な最小 context に留め、T500-2 以降で source snippet allowlist を別設計する。

## 想定外の判断ポイント
既存 `FailedTest.stack` は文字列として absolute path を含み得る。stack 自体は分析価値が高いため drop せず、project root と home directory を placeholder 化してから redaction する。
