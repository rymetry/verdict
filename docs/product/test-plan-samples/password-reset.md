## Intent (PM)
パスワードを忘れたユーザは、登録済 email アドレスを入力するだけで、5 分以内に新パスワードを設定できるべき。

### なぜこれが重要か
- ユーザが domain knowledge と無関係に体験する純粋な UX
- 失敗すると support 問い合わせ流入が即増加 (週間 30 件相当)
- セキュリティ監査の確認項目: token 失効、rate limit、email 検証

### Out of scope
- magic link 経由 sign-in (`magic-link-signin` で別途)
- TOTP / 2FA reset (`mfa-reset` で別途、separate compliance review 経路)
- password 強度ポリシー詳細 (`password-strength` で unit test 推奨)

## Risks (AI 判定)

| Severity | リスク | 根拠 |
|---|---|---|
| **High** | reset token 失効 ロジック | 過去に `expires_at` の timezone bug あり (UTC vs JST 9 時間) |

| **Low** | UI validation 表示 | 入力パスのバリエーション少ない |

## Steps (AI 探索結果)

### 前提
- Test user fixture: `users.standard_b2c` (パスワード `OldPass123!`、登録済 email)
- Email mock: MailHog (port 1025/8025)
- Expected reset URL pattern: `https://app.example.com/reset-password?token=<jwt>`

### Flow
1. **Logout** — clear session, ensure unauthenticated state (`page.context().clearCookies()`)
2. **Navigate to login** — `/login` 遷移
3. **Click "Forgot password"** — `data-testid="forgot-password-link"`
4. **Enter email** — `users.standard_b2c.email` 入力 (`data-testid="reset-email-input"`)
5. **Submit reset request** — `data-testid="reset-submit-button"` クリック
6. **Verify success message** — `data-testid="reset-email-sent-message"` 表示確認
7. **Fetch reset email from MailHog** — `auth-flow` skill の `getLatestEmail(to: email)` 利用
8. **Extract reset token URL** — email body から `https://app.example.com/reset-password?token=...` を regex 抽出
9. **Navigate to reset URL** — extracted URL に遷移
10. **Verify reset form rendered** — `data-testid="new-password-input"` が visible
11. **Enter new password** — `NewPass456!` (rule準拠の強度)
12. **Confirm new password** — `NewPass456!` 再入力 (`data-testid="confirm-password-input"`)
13. **Submit new password** — `data-testid="reset-confirm-button"`
14. **Verify success and redirect** — `data-testid="reset-success-message"` + 自動 `/login` リダイレクト
15. **Login with new password** — 新パスワードで login 成功 (`auth-flow` skill 利用)
16. **Verify old password fails** — logout → 旧パスワードで login → エラー表示
17. **Verify reset token reuse fails** — step 8 の URL に再アクセス → "token used / expired" エラー (High Risk #1)
18. **Cleanup** — `test.afterEach` で user の password を fixture default にリセット

### 並行検証 step (rate limit / High Risk #2)
別 test 関数として:
- 同じ email で 6 回連続 reset request → 4 回目以降に rate limit response (`429`) 確認
