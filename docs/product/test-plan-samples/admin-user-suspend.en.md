## Intent (PM)
A user with administrator privilege should be able to **suspend another user's account** (e.g. for terms-of-service violations).
A suspended user's sessions are invalidated immediately (within 15 seconds), and subsequent login attempts are rejected.

### Why this matters
- Rapid response to abuse is part of our SLA (compliance requirement).
- Complete audit logs are a precondition for regulatory compliance (FISC etc.).
- Verifying privilege boundaries (privilege escalation) is a key item in security audits.

### Out of scope
- Account deletion (`admin-user-delete`; logical-vs-physical deletion debate)
- Bulk suspension (`admin-bulk-suspend`; performance-focused)
- Self-service unsuspension request flow (`user-suspension-appeal`)

## Risks (AI)

| Severity | Risk | Rationale |
|---|---|---|
| **Critical** | Privilege escalation (a normal user can suspend others) | Classic IDOR-style vulnerability; appears repeatedly in OWASP Top 10 |
| **Critical** | Suspension does not propagate; suspended user keeps using the system | Session-cache invalidation bug has caused production incidents (2025-Q4) |
| **High** | Audit log loss / tampering | Direct compliance impact, risk of certification revocation |
| **High** | Suspension affects the admin's own session (admin self-suspends accidentally) | Self-suspension guard required |
| **Medium** | Handling of suspended user's owned resources (project / contract) | Spec confirmation needed (Q1) |
| **Low** | UI status display delay | Real-time display is outside SLA; next page load showing the new status is acceptable |

## Steps (AI exploration result)

### Preconditions
- Admin user fixture: `users.admin` (role: `superadmin`)
- Target user fixture: `users.target_for_suspension` (standard user without ToS violations)
- Audit log endpoint: `/api/audit-log` (admin only, used for observation)
- Session-invalidation timeout: 15 seconds (per SLA)

### Flow A: Happy path (admin suspends target)
1. **Login as admin** — sign in as `users.admin` (uses `auth-flow` skill)
2. **Open second browser context (target user)** — Playwright multi-context: log in as `users.target_for_suspension` in a separate context
3. **Verify target is active** — confirm `/dashboard` is shown in the target context
4. **Admin: navigate to user management** — go to `/admin/users` in the admin context
5. **Search target user** — enter target email in `data-testid="user-search-input"`
6. **Click suspend** — click `data-testid^="suspend-user-button-"` on the target row
7. **Confirm modal** — click `data-testid="suspend-confirm-button"` (reason text `"e2e test reason"`)
8. **Verify admin UI updates** — target row's `data-testid="user-status"` reads `"suspended"`
9. **Verify target session invalidated within 15s** — reload in target context → redirected to `/login` (SLA assertion: `await expect(...).toHaveURL("/login", { timeout: 15000 })`)
10. **Verify target login rejected** — try the original credential → "Account suspended" error (`data-testid="login-error-suspended"`)
11. **Verify audit log entry** — GET `/api/audit-log` → latest entry is `{ action: "suspend", actor: admin.id, target: target.id, reason: "e2e test reason", timestamp: ... }` (Critical Risk #3)

### Flow B: Permission boundary (Critical Risk #1 — IDOR)
1. **Login as standard user** — `users.standard_b2c`
2. **Try to call suspend API directly** — `POST /api/admin/users/<other-user-id>/suspend` via fetch
3. **Verify 403 Forbidden** — `response.status === 403`; body must not leak whether the user exists
4. **Verify other-user is NOT suspended** — confirm in admin context

### Flow C: Self-suspension prevention (High Risk #4)
1. **Login as admin** — `users.admin`
2. **Try to suspend self** — admin's own row's suspend button is disabled or absent in UI
3. **Try via API direct** — `POST /api/admin/users/<self-id>/suspend` → 422 "cannot suspend self"

### Flow D: Audit log immutability (High Risk #3 — partial)
1. After Flow A, **try to delete an audit-log entry directly via DB** (test-only utility; verify no normal API path exists)
2. **Verify**: no audit-log delete endpoint exists (`DELETE /api/audit-log/:id` → 404 not found)
3. **Manual review note**: full tamper-evidence verification deferred to Phase 3 Server's tamper-evident audit log.

### Cleanup
- In `test.afterEach`, unsuspend the target user via the admin context.
- Audit-log traces from tests can remain (test environment uses a separate audit-log schema).
