## Intent (PM)
A user who forgot their password should be able to set a new one within 5 minutes by entering only their registered email address.

### Why this matters
- A pure UX experience independent of domain knowledge.
- Failures immediately spike support inflow (~30 tickets/week).
- Security audit checkpoints: token expiration, rate limiting, email verification.

### Out of scope
- Magic link sign-in (`magic-link-signin`)
- TOTP / 2FA reset (`mfa-reset`, separate compliance review path)
- Detailed password strength policy (recommended as `password-strength` unit tests)

## Risks (AI)

| Severity | Risk | Rationale |
|---|---|---|
| **High** | Reset-token expiration logic | Past timezone bug (UTC vs JST) on `expires_at` |
| **High** | Rate limit bypass | Repeated reset requests for the same email enable email-bombing DoS |
| **Medium** | Cross-account email collisions | B2B structure with same email across tenants; token-to-tenant binding |
| **Low** | UI validation display | Few input variations |

## Steps (AI exploration result)

### Preconditions
- Test user fixture: `users.standard_b2c` (password `OldPass123!`, registered email)
- Email mock: MailHog (port 1025/8025)
- Expected reset URL pattern: `https://app.example.com/reset-password?token=<jwt>`

### Flow
1. **Logout** — clear session, ensure unauthenticated state (`page.context().clearCookies()`)
2. **Navigate to login** — go to `/login`
3. **Click "Forgot password"** — `data-testid="forgot-password-link"`
4. **Enter email** — `users.standard_b2c.email` (`data-testid="reset-email-input"`)
5. **Submit reset request** — click `data-testid="reset-submit-button"`
6. **Verify success message** — `data-testid="reset-email-sent-message"` is shown
7. **Fetch reset email from MailHog** — use `auth-flow` skill's `getLatestEmail(to: email)`
8. **Extract reset token URL** — regex-extract `https://app.example.com/reset-password?token=...` from the email body
9. **Navigate to reset URL** — visit the extracted URL
10. **Verify reset form rendered** — `data-testid="new-password-input"` is visible
11. **Enter new password** — `NewPass456!` (rule-compliant strength)
12. **Confirm new password** — re-enter `NewPass456!` (`data-testid="confirm-password-input"`)
13. **Submit new password** — click `data-testid="reset-confirm-button"`
14. **Verify success and redirect** — `data-testid="reset-success-message"` and auto-redirect to `/login`
15. **Login with new password** — login succeeds with the new password (uses `auth-flow` skill)
16. **Verify old password fails** — logout → login with the old password → error displayed
17. **Verify reset token reuse fails** — revisit the URL from step 8 → "token used / expired" error (High Risk #1)
18. **Cleanup** — reset the user's password to fixture default in `test.afterEach`

### Parallel verification step (rate limit / High Risk #2)
As a separate test:
- Trigger 6 consecutive reset requests for the same email → confirm rate-limit response (`429`) from the 4th onward.
